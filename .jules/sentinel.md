## 2024-11-20 - Fix SSRF DNS Resolution Bypass
**Vulnerability:** The SSRF protection in `src/lib/ssrf.ts` correctly parsed numeric IPv4 and IPv6 string formats (like octal or hex) via `isForbiddenHostname`, but it failed to validate the underlying IP address if an attacker provided a valid-looking string hostname (e.g. `local.attacker.com`) that resolved via DNS to a loopback or internal IP address (like `127.0.0.1` or `169.254.169.254`).
**Learning:** Checking the hostname string is insufficient for robust SSRF protection; domain names can dynamically resolve to internal network spaces, bypassing string-only blacklists.
**Prevention:** Always perform a DNS lookup (e.g., using `dns.lookup({ all: true })`) to resolve the provided hostname to its actual IP addresses, and run the SSRF blocklist validation against every resolved IP address before initiating the outbound connection.
## 2025-03-30 - Fix Redirect-based SSRF Bypass
**Vulnerability:** The SSRF protection in `src/app/api/proxy/route.ts` and `src/app/api/proxy/verify/route.ts` correctly resolved hostnames to IP addresses before performing `fetch` requests, but failed to configure the underlying `fetch` to stop following redirects (default behavior). This allowed an attacker to point the proxy to a safe, attacker-controlled external domain that would then return a `302 Redirect` to an internal metadata or loopback IP (like `169.254.169.254`), completely bypassing the pre-fetch DNS security validation.
**Learning:** Checking the initial hostname or IP string is insufficient if the HTTP client automatically follows redirects. Attackers can leverage open redirects or their own servers to pivot the request *after* validation logic has completed.
**Prevention:** When performing server-side outbound requests where the initial URL is user-controlled or configurable, always enforce `redirect: 'error'` (or manual processing) on the `fetch` operation to prevent silent redirect chaining to internal infrastructure.
